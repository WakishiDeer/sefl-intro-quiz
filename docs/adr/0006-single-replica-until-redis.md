# 0006: インメモリストア期間中はレプリカ数を1に制限する

## Status

accepted

## Context

Azure Container Apps のデプロイ設定で `maxReplicas: 3`（HTTP スケーリングルール付き）が構成されていた。
しかし現在のアーキテクチャでは以下の理由から、複数レプリカ環境では正しく動作しない。

1. **インメモリ状態の分離**: `InMemoryRoomRepository` / `InMemoryQuizRepository` は各プロセスの `Map` に状態を保持するため、レプリカ間で状態が共有されない
2. **Socket.IO アダプターなし**: Redis Adapter を導入していないため、Socket.IO のイベントブロードキャストがレプリカ間で伝播しない
3. **Sticky Session 未設定**: ingress にセッションアフィニティがなく、同一ユーザーが異なるレプリカにルーティングされる可能性がある

結果として、レプリカが2つ以上になると:
- あるレプリカで作成されたルームが別レプリカから見つからない
- Socket.IO のリアルタイム更新が全参加者に届かない
- ルームクリーンアップが各レプリカで独立して動作する

## Decision

1. `maxReplicas` を `1` に設定し、オートスケーリングルールを削除する
2. 防御策として `stickySessions.affinity: 'sticky'` を ingress に追加する（将来 Redis 移行時にレプリカ数を増やした場合の安全策）。ただし Azure Container Apps の sticky session はクッキーベース（クライアント単位）であり、同一ルームの全参加者を同一レプリカに固定する「ルーム単位のアフィニティ」は実現できない。したがって `maxReplicas: 1` が本質的な対策である
3. Redis 移行（tech-spec.md §5.5）が完了するまでレプリカ数を増やさない

## Consequences

### Positive

- インメモリ状態の不整合が発生しない
- Socket.IO イベントが確実に全クライアントに到達する
- 設定変更のみで済み、コード変更不要

### Negative

- 水平スケーリングが不可。高負荷時にスケールアウトできない
- 単一レプリカ障害時にサービスが一時的に利用不可になる（Container Apps が自動復旧するが、インメモリ状態は消失する）

### Risks

- 想定以上のユーザー数（50人超）が同時接続した場合、単一レプリカでは性能不足になる可能性がある。その場合は Redis 移行を前倒しする
